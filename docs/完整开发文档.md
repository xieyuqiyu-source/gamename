## 1. 文档目的

本文档是项目从模板阶段持续开发到 `v1.0.0` 的主依据，用于统一产品范围、玩法规则、工程架构、数据结构、测试标准和版本提交方式。

所有阶段必须遵守以下原则：

1. 每次只推进一个可验证的开发阶段。
2. 每个阶段都必须完成构建、浏览器实机验证和关键交互回归。
3. 每个阶段都要更新本文档、`README.md`、`CHANGELOG.md` 和 `progress.md`。
4. 每个阶段完成后提交 Git、创建语义化版本标签并推送远程仓库。
5. 未通过验收的阶段不得标记为完成版本。
6. 游戏不使用受版权保护的原版美术、音乐、音效或关卡数据。
7. 每个阶段结束必须向用户报告阶段完成项、测试证据、UI 回归结果、Git 版本、总体里程碑进度和下一阶段范围。

## 2. 产品愿景

制作一款保留经典 Breakout 核心手感、同时具有现代视觉反馈和长期成长的网页版打砖块游戏。

玩家通过移动底部挡板，让小球持续反弹并击碎上方砖块。游戏重点不是复杂操作，而是通过高速连击、多球爆发、砖块碎裂、金币散落和道具组合产生持续爽感。

### 2.1 一句话体验

> 控制霓虹挡板，用不断增强的小球击穿砖阵，在爆裂、连击和掉落中完成关卡并持续成长。

### 2.2 核心设计支柱

- 经典易懂：移动挡板、反弹小球、清空砖块。
- 命中爽快：闪光、碎片、拖尾、震屏、波纹和短暂停顿共同强化反馈。
- 掉落驱动：即时道具改变本局节奏，金币支持局外成长。
- 长短结合：20 关主线提供目标，无尽模式提供长期挑战。
- 双端一致：电脑和手机共享规则、关卡和存档结构。
- 无音频设计：完全依靠视觉节奏传达命中、危险和奖励。

## 3. 已确认的设计决策

### 3.1 基础形态

- 采用经典 Breakout 形式。
- 挡板位于画面底部并水平移动。
- 小球持续反弹，击碎上方砖块。
- 游戏主舞台固定为竖屏 `9:16` 逻辑坐标。
- 电脑端居中显示竖屏战场，两侧可展示信息。
- 手机端让战场尽量占满可用视口。

### 3.2 内容结构

- 主线共 20 关，分为 4 章，每章 5 关。
- 第 5、10、15、20 关为 Boss 关。
- 通关第 10 关后解锁无尽模式。
- 主线关卡有三星评价。
- 无尽模式记录历史最高分、最高波次和最高连击。

### 3.3 生命规则

- 每关默认 3 条生命。
- 小球全部落出底部时扣除 1 条生命。
- 掉球后砖块破坏进度保留。
- 掉球后清空当前即时道具效果，只保留一颗待发射球。
- 剩余生命为 0 时关卡失败。
- 永久升级可以额外增加 2 条初始生命，最终最多 5 条。

### 3.4 砖块规则

- 普通砖块：1 点耐久，一次命中即破碎。
- 强化砖块：2～3 点耐久，颜色与普通砖明显不同。
- 强化砖每次受击后改变亮度、颜色或裂纹状态。
- 最后一击产生更强的碎裂和掉落反馈。
- 后期允许加入不可破坏砖、移动砖、护盾砖和机关砖，但必须分章节逐步引入。

### 3.5 即时道具

第一批只加入 5 种正面道具：

| 道具 | 效果 | 初始建议值 | 叠加方式 |
| --- | --- | --- | --- |
| 多球 | 当前小球分裂 | 每颗分裂为 3 个角度 | 最多 12 颗球 |
| 挡板变长 | 提高挡板宽度 | 每层增加 22% | 最多 2 层，之后刷新时间 |
| 穿透球 | 小球穿过可破坏砖 | 8 秒 | 刷新持续时间 |
| 小球减速 | 降低全部小球速度 | 10 秒，速度乘 0.72 | 刷新持续时间 |
| 挡板激光 | 挡板可发射双激光 | 10 秒，带射击冷却 | 刷新持续时间 |

规则：

- 不同种类道具可以同时生效。
- 同类持续型道具重复拾取时刷新时间，不无限叠加。
- 所有即时效果在掉球后清除。
- 具体持续时间和数值在浏览器试玩后调整。

### 3.6 掉落与接取

- 即时道具以发光胶囊形式垂直下落。
- 即时道具必须由挡板接住，漏掉后消失。
- 金币从砖块位置散落，带轻微弹跳与旋转。
- 金币靠近挡板时受到磁力吸附。
- 永久升级“金币磁力范围”只影响金币，不影响即时道具。
- 屏幕同时存在的掉落物必须设置数量上限，避免性能和可读性失控。

### 3.7 连击与分数

- 连续命中砖块会增加连击数。
- 命中砖块后刷新连击保持时间。
- 连击中断条件：全部小球掉落，或超出保持时间未击中砖块。
- 连击提高得分倍率、金币收益和即时道具掉率。
- 高连击增强拖尾、粒子、色彩、画面脉冲和震屏幅度。
- 连击强度必须有视觉分层，不能只显示数字。

初始倍率建议：

| 连击区间 | 分数倍率 | 金币倍率 | 道具掉率修正 |
| --- | ---: | ---: | ---: |
| 0～4 | ×1.0 | ×1.0 | +0% |
| 5～14 | ×1.25 | ×1.1 | +1% |
| 15～29 | ×1.5 | ×1.2 | +2% |
| 30～49 | ×2.0 | ×1.35 | +3% |
| 50+ | ×3.0 | ×1.5 | +5% |

以上均为初始调试值，不是最终平衡承诺。

### 3.8 发球与反弹

- 每次生命开始时，小球附着在挡板上方。
- 点击、轻触或按空格发球。
- 初始发射角约为 45°，左右方向可交替或受挡板位置影响。
- 击中挡板左侧时向左反弹，击中右侧时向右反弹。
- 击中位置越靠近挡板边缘，水平速度分量越大。
- 挡板自身移动方向和速度会轻微修正反弹角度。
- 必须限制最小垂直速度，防止小球近水平无限往返。
- 必须限制最高速度，确保移动端仍可操作。

### 3.9 Boss 规则

- Boss 由巨型核心、护盾砖或攻击模块组成。
- 第 1 章 Boss：移动核心和护盾砖。
- 第 2 章 Boss：增加可破坏攻击模块。
- 第 3 章 Boss：加入基础弹幕。
- 第 4 章 Boss：移动、护盾、弹幕和阶段变化结合。
- Boss 弹幕命中挡板时优先消耗护盾。
- 护盾耗尽后再次被弹幕命中，扣除 1 条生命。
- 损失生命后恢复部分护盾并重新发球。
- Boss 护盾上限可以通过局外永久升级提升。

### 3.10 星级评价

每关最多获得三星：

1. 成功通关获得第一星。
2. 通关时至少剩余 2 条生命获得第二星。
3. 达到该关目标分数或连击要求获得第三星。

星级用于：

- 解锁章节奖励。
- 展示关卡完成度。
- 为后续挑战目标提供依据。

### 3.11 永久升级

金币可以在关卡外用于升级：

| 升级项 | 等级上限 | 作用 |
| --- | ---: | --- |
| 挡板宽度 | 10 | 提高基础挡板宽度 |
| 道具掉率 | 10 | 提高即时道具基础掉率 |
| 金币收益 | 10 | 提高结算金币与掉落金币价值 |
| 金币磁力范围 | 10 | 提高金币被吸附的距离和力度 |
| 连击保持时间 | 10 | 延长连击中断前的宽限时间 |
| Boss 护盾上限 | 5 | 提高 Boss 战挡板护盾值 |
| 初始生命 | 2 | 将默认 3 条生命提高至最多 5 条 |

升级原则：

- 前期提升明显，后期收益递减。
- 每项设置等级上限。
- 升级费用逐级增长。
- 在关卡外可以免费重置，返还全部已投入金币。
- 主线关卡难度随章节增长，不能让永久成长完全替代操作。

### 3.12 音频

- 游戏不加入背景音乐。
- 游戏不加入音效。
- 所有反馈必须在静音环境下清晰可读。
- 设置中无需音量选项，但需要特效强度和震屏开关。

## 4. 游戏流程

### 4.1 首次进入

1. 加载本地存档。
2. 没有存档则创建 `saveVersion = 1` 的默认存档。
3. 展示标题页和“开始游戏”。
4. 进入主界面后默认开放第 1 关。

### 4.2 主线流程

```text
标题页
  → 主界面
  → 关卡选择
  → 关卡准备
  → 游戏中
  → 胜利/失败结算
  → 保存进度
  → 返回关卡选择或重试
```

### 4.3 无尽模式流程

```text
通关第 10 关
  → 解锁无尽模式
  → 选择当前永久升级配置
  → 连续波次挑战
  → 砖块耐久、移动速度和机关密度增长
  → 生命耗尽后结算最高分、最高波次、金币
```

### 4.4 暂停与恢复

- 桌面端按 `Esc` 或点击暂停按钮。
- 移动端切换后台、页面失焦或触发系统手势时自动暂停。
- 暂停时停止物理、计时器、掉落物和特效时间。
- 恢复前提供短暂倒计时，避免立即掉球。

`v0.4.0` 已实现：运行中失焦或页面进入后台会自动转为 `paused`；暂停期间挡板、物理、道具计时、掉落物、命中停顿和全部特效时间均冻结；玩家确认后进入 `countdown`，完成 3 秒倒计时才恢复 `playing`。

## 5. 页面与界面结构

### 5.1 页面状态

第一版不要求 Vue Router，多页面状态可由根界面状态机管理：

- `title`：标题页。
- `home`：主界面。
- `levelSelect`：关卡选择。
- `upgrades`：永久升级。
- `playing`：游戏中。
- `paused`：暂停。
- `result`：结算。
- `endless`：无尽模式准备或结算。
- `settings`：特效、震屏、操作设置。

### 5.2 桌面布局

- 中央：竖屏 `9:16` Canvas 游戏战场。
- 左侧：关卡、生命、目标、连击和分数。
- 右侧：道具状态、金币、暂停和操作提示。
- 浏览器高度不足时按比例缩放战场，不能产生内部滚动。

### 5.3 移动布局

- Canvas 尽量占据可用屏幕。
- 生命、分数、连击等叠加在安全区内。
- 避开刘海、圆角和浏览器底部工具栏。
- 触摸拖动区域覆盖整个战场宽度，不要求手指必须按住挡板。
- 关键按钮触控尺寸不小于 44 CSS 像素。

## 6. 输入设计

### 6.1 电脑端

- 鼠标横向移动控制挡板。
- `A` / `D` 或 `←` / `→` 控制挡板。
- 鼠标点击或 `Space` 发球。
- 激光状态下，点击或 `Space` 发射激光。
- `Esc` 暂停或返回。
- `F` 切换全屏。

### 6.2 手机端

- 横向拖动控制挡板。
- 轻触发球。
- 激光状态下，轻触战场发射激光；拖动不触发误射。
- 浏览器失焦自动暂停。
- 激光生效时保留独立暂停按钮，不让主动射击占用唯一控制入口。
- 标题与结算状态提供紧凑存档状态和两步确认重置入口。

### 6.3 输入抽象

输入层统一输出：

```js
{
  targetX: 0,
  moveAxis: 0,
  launchPressed: false,
  actionPressed: false,
  pausePressed: false,
}
```

游戏逻辑不得直接依赖鼠标或触摸事件，避免双端规则分叉。

## 7. 物理与碰撞规范

### 7.1 逻辑坐标

- 推荐逻辑分辨率：`540 × 960`。
- 所有游戏对象使用逻辑坐标，不直接使用 CSS 像素。
- Canvas 根据设备像素比提高渲染清晰度，但物理坐标保持不变。

### 7.2 固定时间步

- 物理更新使用固定步长，建议 `1 / 120` 秒。
- 渲染可以跟随 `requestAnimationFrame`。
- 自动测试通过 `window.advanceTime(ms)` 推进确定性时间。
- 单帧最大补偿步数必须限制，避免页面恢复后物理爆炸。

### 7.3 高速碰撞

- 小球高速移动时不能只检测终点位置。
- 第一阶段可采用子步进检测。
- 速度提高后应升级为 swept circle / ray cast 思路，避免穿砖。
- 同一物理步内要避免对同一砖块重复结算多次。

### 7.4 碰撞优先级

1. 边界。
2. 挡板。
3. 砖块和 Boss 部件。
4. 掉落物与挡板。
5. 弹幕与挡板。

穿透球命中砖块后不反向，但仍触发伤害、连击和特效。

## 8. 视觉反馈规范

### 8.1 命中反馈层级

普通命中：

- 砖块快速增亮。
- 4～8 个小粒子。
- 小范围波纹。
- 小球拖尾短暂加粗。

破碎命中：

- 砖块拆分为碎片。
- 12～20 个粒子。
- 轻微画面震动。
- 2～4 帧命中停顿。

强化砖最后一击或高连击：

- 更强闪光与冲击环。
- 画面色差或局部辉光增强。
- 震屏幅度随连击提升，但有上限。
- 掉落物生成时显示短促光柱。

### 8.2 可访问性

- 提供震屏开关。
- 提供特效强度：低、中、高。
- 颜色不能作为唯一信息，强化砖同时使用边框、纹理或耐久标记。
- 高强度闪光必须限制频率和亮度，避免频闪风险。

### 8.3 性能预算

- 桌面目标 60 FPS。
- 中端手机目标 60 FPS，低端设备最低 30 FPS。
- 粒子采用对象池，避免频繁创建和回收对象。
- 同屏粒子默认上限建议 600，低特效模式上限 180。
- 同屏小球上限 12。
- 同屏掉落物上限建议 24。

## 9. 关卡规划

### 9.1 第一章：霓虹起点（1～5）

- 目标：教授移动、发球、普通砖、强化砖和基础掉落。
- 第 1 关：简单矩形阵列，验证核心手感。
- 第 2 关：间隔阵列，引导控制反弹角度。
- 第 3 关：首次出现 2 点强化砖。
- 第 4 关：集中掉落与连击挑战。
- 第 5 关 Boss：移动核心 + 护盾砖。

### 9.2 第二章：脉冲回路（6～10）

- 目标：引入移动砖与攻击模块。
- 第 10 关 Boss：核心 + 可破坏攻击模块。
- 通关第 10 关后解锁无尽模式。

### 9.3 第三章：过载矩阵（11～15）

- 目标：引入 3 点强化砖、机关砖和基础弹幕。
- 第 15 关 Boss：多阶段护盾 + 基础弹幕。

### 9.4 第四章：终极奇点（16～20）

- 目标：组合全部规则并提高节奏。
- 第 20 关 Boss：移动核心、护盾、攻击模块、弹幕和阶段变化。

### 9.5 关卡数据格式

```js
{
  id: 1,
  chapter: 1,
  name: '初次折射',
  rows: 8,
  columns: 9,
  layout: [],
  targetScore: 12000,
  targetCombo: 20,
  baseDropRate: 0.08,
  boss: null,
}
```

关卡必须数据驱动，不允许将砖块位置散落硬编码在组件中。

## 10. 工程架构

### 10.1 技术边界

- Vue：页面、菜单、HUD、弹窗和设置。
- Pinia：跨页面状态、存档数据、关卡进度、升级和设置。
- Canvas 2D：游戏实体、物理、粒子、拖尾和战斗特效。
- localStorage：正式存档。
- 不依赖后端，不要求登录，不上传用户数据。

### 10.2 推荐目录

```text
src/
├── components/          # Vue UI 组件
├── config/              # 平衡参数、道具、升级、关卡配置
├── engine/
│   ├── GameEngine.js    # 游戏循环和场景生命周期
│   ├── physics.js       # 碰撞与运动
│   ├── renderer.js      # Canvas 绘制
│   ├── input.js         # 鼠标、键盘、触摸统一输入
│   ├── effects.js       # 粒子、震屏、命中停顿
│   └── pools.js         # 对象池
├── entities/            # Ball、Paddle、Brick、Drop、Projectile
├── levels/              # 20 关与无尽生成规则
├── stores/
│   ├── app.js           # 页面模式和全局 UI
│   ├── save.js          # 存档和版本迁移
│   ├── progression.js   # 星级、金币、升级、解锁
│   └── run.js           # 当前关卡运行状态
├── utils/               # 数学、格式化、随机数
├── App.vue
├── main.js
└── style.css
```

### 10.3 状态分层

持久状态：

- 金币。
- 关卡解锁。
- 每关最高星级和最高分。
- 无尽模式最高纪录。
- 永久升级等级。
- 设置。

临时运行状态：

- 当前生命。
- 当前分数和连击。
- 当前小球、砖块、掉落物和特效。
- 当前道具持续时间。
- Boss 当前阶段。

Canvas 高频实体不应每帧写入 Pinia，避免响应式开销。Pinia只同步对 UI 和存档有意义的低频摘要。

## 11. 存档规范

### 11.1 存储键

正式存档键：

```text
gamename:save
```

### 11.2 存档结构

```js
{
  saveVersion: 1,
  profile: {
    createdAt: 0,
    updatedAt: 0,
  },
  currency: {
    coins: 0,
  },
  campaign: {
    highestUnlockedLevel: 1,
    levelRecords: {
      1: { stars: 0, highScore: 0, bestCombo: 0 },
    },
  },
  endless: {
    unlocked: false,
    highScore: 0,
    highestWave: 0,
    bestCombo: 0,
  },
  upgrades: {
    paddleWidth: 0,
    itemDropRate: 0,
    coinBonus: 0,
    magnetRange: 0,
    comboGrace: 0,
    bossShield: 0,
    extraLife: 0,
  },
  settings: {
    effectQuality: 'high',
    screenShake: true,
    controlMode: 'auto',
  },
}
```

`v0.4.0` 在不改变 `saveVersion = 1` 主结构的前提下，为关卡记录补充 `attempts`、`clears`、`bestLives`、`lastPlayedAt`，并在 `campaign.lastResult` 保存最近一次结算摘要。新增字段全部具有默认值，旧的正式结构可安全归一化。

### 11.3 保存时机

- 关卡胜利或失败结算后。
- 购买或重置升级后。
- 设置变更后。
- 无尽模式结算后。
- 页面关闭前可补充保存，但不能只依赖 `beforeunload`。

### 11.4 版本迁移

- 每次修改存档结构必须提升 `saveVersion`。
- 提供从旧版本到新版本的逐级迁移函数。
- 无法识别的字段忽略，不能直接使整个存档失效。
- 解析失败时保留原始字符串备份，再创建新存档。

## 12. 测试策略

### 12.1 必备测试接口

- `window.render_game_to_text()`：输出当前可交互状态。
- `window.advanceTime(ms)`：确定性推进固定时间步。
- 可选的开发调试入口只在开发环境启用。

### 12.2 文本状态至少包含

- 页面模式。
- 当前关卡和关卡状态。
- 挡板位置、宽度和护盾。
- 所有小球的位置、速度和效果。
- 剩余砖块和耐久。
- 掉落物类型、位置和速度。
- 生命、分数、连击、金币。
- 道具剩余时间。
- 胜利、失败和暂停标记。

### 12.3 每阶段浏览器验证

1. 使用项目 Vite 开发服务器启动。
2. 使用规定的网页游戏 Playwright 客户端执行操作。
3. 读取 `render_game_to_text` 状态。
4. 检查最新 Canvas 截图。
5. 检查控制台和页面错误。
6. 对核心操作完成“输入 → 中间状态 → 结果”的整链路验证。
7. 修复问题后重新运行完整回归。

### 12.4 核心测试矩阵

| 范围 | 必测内容 |
| --- | --- |
| 挡板 | 键盘、鼠标、触摸、边界限制、不同帧率 |
| 小球 | 发射、墙面反弹、挡板角度、最低垂直速度、掉球 |
| 砖块 | 普通砖、强化砖、多次受击、同帧重复碰撞 |
| 生命 | 扣命、重置、进度保留、最终失败 |
| 道具 | 掉落、漏接、拾取、叠加、刷新、掉球清空 |
| 多球 | 分裂、上限、部分掉落、全部掉落 |
| 金币 | 生成、弹跳、磁力、结算、永久保存 |
| 连击 | 增加、保持、中断、倍率和视觉层级 |
| 关卡 | 胜利、失败、星级、解锁、重试 |
| Boss | 护盾、部件、弹幕、阶段、胜利 |
| 存档 | 自动保存、刷新恢复、版本迁移、损坏恢复 |
| 适配 | 桌面 16:9 容器、手机竖屏、安全区、全屏 |

### 12.5 UI 与画面强制回归

UI 和画面是本项目的最高优先级之一。每个开发阶段都必须进行持续性 UI 回测，不能只依据代码或文本状态判断完成。

每阶段至少检查以下视口：

| 场景 | 建议视口 | 检查重点 |
| --- | --- | --- |
| 标准桌面 | 1440 × 900 | 战场与侧栏比例、信息层级、留白 |
| 小型桌面 | 1280 × 720 | 首屏完整性、按钮换行、纵向溢出 |
| 标准手机 | 390 × 844 | 竖屏战场、触控尺寸、安全区 |
| 窄屏手机 | 360 × 800 | 文本截断、HUD 重叠、底部工具栏影响 |

每次 UI 回归必须同时检查：

1. 完整页面截图，而不仅是 Canvas 截图。
2. Canvas 游戏画面截图，确认实际渲染对象完整可见。
3. 标题、按钮、数值、标签是否出现换行、截断或遮挡。
4. 信息层级是否清晰，主操作是否在 2 秒内可识别。
5. 霓虹发光是否影响文字和砖块轮廓的可读性。
6. 动态状态：默认、悬停、按下、禁用、暂停、胜利、失败。
7. 高连击、多球、密集粒子和大量掉落时是否仍然可读。
8. 桌面鼠标和移动触控反馈是否一致。
9. 控制台是否产生布局、资源或渲染错误。

UI 问题处理原则：

- 发现明显视觉缺陷后，不允许仅记录为后续事项；应在当前阶段修复并重新截图。
- 每次只调整一个主要视觉变量，再进行对比验证。
- 视觉优化不得破坏 `render_game_to_text` 与实际画面的一致性。
- 正式版本标签必须以最新 UI 截图已人工检查为前提。

## 14. Git 与版本提交规范

### 14.1 分支

- `main`：始终保持可构建、可运行。
- 大阶段可以使用功能分支，阶段验收后合并。
- 不将未验证的实验代码标记为版本。

### 14.2 提交信息

推荐格式：

```text
feat: 完成核心物理原型
fix: 修复高速小球穿砖
docs: 更新 v0.3.0 道具验收记录
test: 补充多球掉落回归场景
chore: 发布 v0.4.0
```

### 14.3 阶段发布流程

1. 完成阶段范围。
2. 运行 `pnpm build`。
3. 启动开发服务器。
4. 在 Chrome 中完成自动测试与截图检查。
5. 检查控制台错误。
6. 更新 README、开发文档、CHANGELOG 和 progress。
7. 更新 `package.json` 版本。
8. 提交全部阶段变更。
9. 创建带注释标签，例如 `git tag -a v0.2.0`。
10. 推送分支和标签到远程仓库。
11. 输出阶段进度报告，至少包含“本阶段完成”“验证结果”“当前版本”“总体进度”“下一阶段”五部分。

## 15. 完成定义

一个阶段只有同时满足下列条件才算完成：

- 功能范围全部实现。
- 关键交互完成自动化与手动浏览器验证。
- 最新 Canvas 截图通过视觉检查。
- `render_game_to_text` 与画面状态一致。
- 无新增控制台错误。
- 生产构建成功。
- 文档和变更记录更新。
- Git 提交、版本标签和远程推送完成。
